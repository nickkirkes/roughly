**Word-cap analysis (the constraint):** `agents/doc-writer.md` is at 649/650 words (`wc -w` confirmed pre-edit). The minimum useful rationale that satisfies AC2's "parent guaranteed because [reason]" form ("setup creates file; Edit-only — no mkdir-p needed") is ~10 words. Any net-positive word addition at the L22 bullet would push the file over the 650 cap and fail verify-all.sh's hard-enforcing check (`.claude/hooks/verify-all.sh:27–30`). Recovering 10+ words of slack from other prose in doc-writer.md would violate AC5's verify: "`git diff` inspection shows only single-line additions of `mkdir -p` or rationale comments at the Write sites; no surrounding prose modified" — subtractions are not single-line additions.

**Safety analysis (why the carve-out is safe):** The doc-writer agent uses `Edit` (not `Write`) to mutate `.roughly/known-pitfalls.md` (see L39 "Invoke `Edit` per file in multi-file dispatch"). The Edit tool fails loudly when the target file is missing — no silent-failure mode exists, which is the original silent-failure risk that `mkdir -p` instructions guard against at `Write` sites. The actual safety property (file pre-exists when doc-writer is dispatched; `/roughly:setup` Step 5b creates it before any pipeline that dispatches doc-writer can run) is preserved regardless of where the rationale prose lives. Documenting in the audit report makes the safety reasoning canonical and discoverable without breaching the word cap.

**Pre-flight verify:** Confirm `wc -w agents/doc-writer.md` returns 649 (re-verifies cap proximity before deciding the carve-out is justified — if doc-writer.md somehow has slack at implement-time, T2 would revert to a small inline addition and the carve-out classification would not be used).
