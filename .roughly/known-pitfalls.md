# Known Pitfalls

Project: Roughly
Domain: Roughly is a Claude Code plugin that implements gated development pipelines with subagent-per-task execution.

Pitfalls discovered through development. Updated by `/roughly:build` and `/roughly:fix` wrap-up stages.

---

## Domain-Specific

- **Agent files are plugin-shipped, not project-installed.** All 7 agents are loaded via `subagent_type` (e.g., `roughly:code-reviewer`) from the plugin cache. The upgrade skill must never classify agent files as "New" or offer to copy them to `.claude/agents/`. The only valid agent-related check during upgrade is preamble drift on pre-existing `.claude/agents/` files.

- **Plan mode hijack of build/fix is now blocked by S1 enforcement (ADR-009).** When `/roughly:build` or `/roughly:fix` is invoked with plan mode active, the `UserPromptSubmit` hook at `.claude/hooks/plan-mode-gate.sh` reads `permission_mode` from stdin and emits a `decision:"block"` JSON, preventing the prompt from reaching the model. The historical silent failure (plan-mode's Phase 1–5 workflow substituting for Stages 1–4 and silently skipping `/roughly:review-plan`) is no longer reachable when the hook is registered — see ADR-009 and `docs/planning/spikes/plan-mode-detection-findings.md` for the full evidence trail and the spike-doc event-name correction (`UserPromptExpansion` → `UserPromptSubmit`). Recovery if you see the block: exit plan mode (Shift+Tab) and re-invoke. Hook is installed by `/roughly:setup` Step 5d for new projects; existing projects on v0.1.4 must re-run setup or run `/roughly:upgrade` to install. If a user's `.claude/settings.json` already has a `UserPromptSubmit` entry, setup will not overwrite it — verify manually that plan-mode protection is in place.

## Data & State

- **CLAUDE.md as the source of truth for verify-all commands has two known failure modes.** Teams with mandated CLAUDE.md formats may not permit Roughly's Commands table, and third-party agents (claude-mem and others) that rewrite CLAUDE.md programmatically can clobber it. Session-length compaction is *not* a failure mode — skills explicitly `Read` CLAUDE.md at runtime per ADR-006, so the disk file is authoritative regardless of what was autoloaded. If breakage is reported, the clean fix is an additive `.roughly/commands.md` fallback that skills check first, falling back to CLAUDE.md — strictly additive, no migration. Stay on CLAUDE.md until that happens to avoid a two-source-of-truth mental model.

## Build & Deploy

- **Pre-implementation review (review-epic, review-plan) catches design issues but not always execution bugs.** A spec can pass two `/roughly:review-epic` iterations and a `/roughly:review-plan` pass yet still contain logic bugs that only execution-tracing catches in code review. Example: the S2.2 v0.1.4 migration step had "marker write before conflict check" ordering that made the conflict-prompt branch unreachable on first run — pre-implementation reviewers read the steps as a bullet list; only the Stage 6 code-reviewer noticed by tracing execution order. Don't treat spec-faithfulness as a substitute for code review. Stage 6 catches what Stages 2–4 miss.

- **Grep-based ACs are authoritative over a spec's line-enumeration tables.** When an epic specifies "zero `\bRuckus\b` matches in skills/" alongside an enumerated list of lines to edit, the AC is the contract — the line list is a (sometimes incomplete) implementation hint. Plans should reconcile the AC against `rg -n` output during plan-write and add any missing lines to the substitution table. S2.2 had two reconciled gaps: setup/SKILL.md L40 (mixed-content line, caught by plan-reviewer) and L3 frontmatter descriptions (caught at plan-write time). Pattern: trust the regex, not the line number.

- **`set -euo pipefail` combined with unguarded parser tools silently opens safety-gate hooks.** When a hook script's job is to gate dangerous operations — block a tool call, halt a pipeline — `set -e` inverts the safety posture: any non-zero exit from any subprocess kills the script before the gate logic runs. In E03.S1's first draft of [`.claude/hooks/plan-mode-gate.sh`](../.claude/hooks/plan-mode-gate.sh), `set -euo pipefail` was active and `jq` was called unguarded to extract `permission_mode` from stdin JSON. Malformed JSON causes `jq` to exit non-zero; `set -e` then kills the script; no block JSON is emitted; Claude Code receives `exit 0` with no output — silently passing the prompt through the gate it was meant to close. The silent failure was caught by the silent-failure-hunter agent at Stage 6 review, before commit. Resolution: drop `set -e` (keep `-uo pipefail`), wrap parser tools in conditional guards (`if jq_out=$(... | jq ...); then ... fi`), and provide explicit fail-closed branches at every parsing failure point — `jq` → `python3` → `printf` fallback chain with no silent skip. **Pattern:** in any safety-gate script, `set -e` means "fail loudly and exit non-zero" — but for a gate that signals by emitting JSON, that means "fail silently and open the gate." Drop `set -e`, keep `set -uo pipefail`, and audit every parser-tool call for fail-closed alternatives. Reference: [docs/adrs/ADR-009-plan-mode-detection.md](../docs/adrs/ADR-009-plan-mode-detection.md); commit c598ef6.

- **`set -e` + command-substitution-in-assignment silently masks CI assertion failures.** When a script under `set -euo pipefail` does `OUT="$(timeout N cmd 2>&1)"` and the inner command exits non-zero (timeout 124, auth failure, budget breach, etc.), the script dies at the assignment line — before any `if grep ...` assertion against `$OUT` can fire. The intended FAIL-emit branch is unreachable; the only signal is a generic non-zero exit with no diagnostic text, indistinguishable from a script-died-mid-execution failure. In E03.S11b-1 (commit `3f6ed43`), both the smoke (`grep -qx "ok"`) and plugin-load (`grep -q "roughly:setup"`) assertions in `scripts/ci-dogfood.sh` were initially written with the naive `OUT="$(...)"` pattern; both reviewers (code-reviewer + silent-failure-hunter) flagged the mode independently, and `bash -c 'set -e; X=$(false); echo still-here'` confirmed empirically that the script dies before the echo. **Fix:** capture the exit code explicitly with `... && OUT_EXIT=0 || OUT_EXIT=$?`, then branch on `124` (timeout — distinct "timed out" message), other-non-zero (failure with output dump), and only run the grep when EXIT=0. **Pattern:** in any CI / assertion bash script, do not assign command substitution directly under `set -e` when distinct failure diagnostics matter. The `&& X=0 || X=$?` idiom is concise, matches the convention already used in `.github/workflows/dogfood.yml`'s auth-failure step, and produces a human-friendly diagnostic ladder ("FAIL — timed out" / "FAIL — exited N" / "FAIL — assertion mismatch") instead of a silent script death.

## Skill & Agent Authoring

- **LLM agent conditionals need explicit failure-handling clauses.** Agent prompts are prose instructions, not typed branches — implicit failure paths are filled by the model's heuristics, sometimes safely (skip) and sometimes unsafely (estimate, infer, fall back). In E03.S3, adding two conditional sub-bullets to `agents/doc-writer.md` produced 3 P1 silent-failure findings because neither bullet specified what to do when its precondition was not met. When reviewing any addition of conditional logic to an agent prompt, treat every precondition as a separate failure-mode question: "If this precondition fails, what does the LLM do?" Specify the skip path explicitly. Applies to all agents, especially `doc-writer.md`, `investigator.md`, `discovery.md`, and any new agents added in v0.1.5+.

- **LLM-interpreted conditionals must reference observable signals, not abstract categories.** When a skill's prose tells the orchestrator to act "if X," X must be expressible as an observable artifact — a string in command output, a file path, an exit code — not as an inferred category like "the test command" or "the user's intent." In E03.S10, the auto-fix-cap conditional was first written as "if the failing check was the test command, escalate after attempt 2." Two reviewers (code-reviewer + silent-failure-hunter) independently flagged the same issue: the orchestrator has no inherent knowledge of whether a command is the test runner versus the type-checker; it only observes command output. A confident misclassification silently applies cap 4 instead of cap 2, defeating the intent. Fix: reframe as "if the failure output contains assertion errors or test-runner output." Anti-examples: "if the failing check was the type-check command" (abstract category), "if the user wants to proceed" (intent inference). Concrete forms: "if the output contains X," "if the user's response begins with the literal token 'override'," "if `package.json` contains a `scripts.test` field." This pitfall is complementary to the one above — grounding (this entry) + explicit-failsafe (above entry). Reference: commit `f60056a`; E03.S10 convergent review flag.

- **Conflict-prompt branches must explicitly enumerate state cleanup for every outcome, including no-op paths.** When a skill writes state (a file, a `settings.json` entry, a `.roughly/workflow-upgrades` record) *before* showing a multi-option prompt, every branch — including `keep` and `decline` — must explicitly say what happens to that pre-written state. LLMs reading skill prose interpret silence as "do nothing," which leaves orphan files or stale records that contradict the user's displayed installation status. Concrete origin: S03.2 (`skills/setup/SKILL.md` Step 5d Branch 4) wrote `.claude/hooks/verify-all.sh` before showing a keep/replace/merge/decline conflict prompt. The original `keep` branch said "no-op" — leaving the file on disk with no record entry; the original `decline` branch said "delete if already written" but the file was always already written by that point, making the conditional language misleading. Fix: every branch must explicitly state filesystem cleanup, `settings.json` cleanup, and `workflow-upgrades` record cleanup, with `rm`-failure error handling surfaced to the user. **Pattern:** when authoring any multi-branch prompt in a skill, audit each branch against every pre-prompt side effect and confirm all are explicitly resolved — don't rely on the model to infer cleanup. (2026-05-04)

- **Tool availability must be checked before any file writes when both are required for a complete install.** If a skill needs an external tool (e.g., `jq`, `python3`) AND a file write to complete an install step, check tool availability first. Writing the file first and checking the tool afterward leaves an orphan file when the tool is absent — half-installed state with no rollback path and no error message at the point of divergence. Concrete origin: S03.2 Stage 8 stop-hook-v1 install in `skills/build/SKILL.md` and `skills/fix/SKILL.md` instructed: read template → substitute placeholders → write `.claude/hooks/verify-all.sh` → update `.claude/settings.json` via `jq`. When `jq` was unavailable, the hook file landed on disk but was never registered in `settings.json` — the filesystem and settings were silently out of sync with no warning. Fix: reorder so `jq` availability is confirmed before any file writes; if absent, write nothing and surface a clear warning. **Pattern:** enumerate all tool prerequisites at the top of any install branch — confirm tools are available and target directories are writable before mutating state. This is the transactional "validate inputs before mutations" rule applied to skill orchestration. (2026-05-04)

- **`setup/SKILL.md`'s pre-flight migration check is a soft-abort and must NOT be normalized to the canonical hard-abort form.** As of E03.S4 (commit `1062432`), 8 of 9 skills have a `.ruckus/` pre-flight migration check. The other 7 (audit-epic, build, fix, review, review-plan, review-epic, verify-all) use a canonical bold-paragraph hard-abort — byte-identical across all 7. `setup/SKILL.md` L39–40 uses a deliberate two-line prose+blockquote form with a "(proceed anyway / abort)" soft-override, because setup is the install skill: hard-aborting from setup would prevent legitimate install-time recovery from legacy state. A future contributor running `rg -c "Legacy \`.ruckus/\` state detected" skills/*/SKILL.md` will correctly see 8 matches and may "fix" setup's divergent form to match the canonical one — silently removing the soft-override. **Pattern:** byte-identity drift checks must be scoped to the 7 hard-abort skills only, not all 8. Canonical-form verification: `rg "Legacy \`.ruckus/\` state detected" skills/audit-epic/SKILL.md skills/build/SKILL.md skills/fix/SKILL.md skills/review/SKILL.md skills/review-plan/SKILL.md skills/review-epic/SKILL.md skills/verify-all/SKILL.md | sort -u | wc -l` must output `1`. Setup is verified separately by reading L39–40 and confirming the soft-abort "(proceed anyway / abort)" form is intact.

- **`Glob` has no mtime field — use `ls -lt` when skill instructions require sort-by-modification-time or displayed timestamps.** A skill body that says "Use `Glob` for `<pattern>`. Sort results by modification time descending." gives the LLM a signal it cannot observe: `Glob` returns matches in alphabetical/filesystem order with no timestamp field. Any subsequent instruction that references a modification date (e.g., "emit modified YYYY-MM-DD HH:MM") has nowhere to read the value from — the model will either hallucinate a timestamp or emit a blank. This is an "ungrounded conditional": the instruction references a signal the named tool does not produce. Concrete origin: E03.S8 `skills/help/SKILL.md` initial draft; caught at Stage 6 review. **Fix:** when a skill needs mtime, the instruction must name a command whose output carries the timestamp: `ls -lt <pattern> 2>/dev/null`, `stat`, or `find -printf '%T@ %p\n'`. Match the required signal to a command that observably produces it. This is a concrete named-tool instance of the observable-signals rule (see "LLM-interpreted conditionals must reference observable signals" above).

- **Interactive re-prompt loops in utility skills must have a max-retry guard.** A skill body (in a `disable-model-invocation: false` skill) that says "If the user's response does not match any listed option, re-display the list and re-ask" with no termination condition gives the LLM no upper bound. Confused users entering bad input loop the skill indefinitely, blocking the session. Pipeline skills (`disable-model-invocation: true`) already carry explicit caps (e.g., "max 2 attempts; then escalate") because the retry-loop risk is recognized there. Utility skills (`setup`, `upgrade`, `help`) inherited the prompt-and-react pattern without inheriting the cap discipline. Concrete origin: E03.S8 `skills/help/SKILL.md` disambiguation prompt; caught at Stage 6 review. **Fix:** any user-input loop in a utility skill must include a max-retry count and a graceful-exit emit when the cap is hit. Pattern: "After N invalid responses, emit `<recovery message>` and end the step without picking." The exit must continue the skill, not abort — utility skills are by contract non-blocking.

- **`$ARGUMENTS` flag detection has a Goldilocks zone: substring matching is too loose, first-token matching is too strict, standalone-token matching is right.** When skill prose detects a CLI-style flag in `$ARGUMENTS`, both naive forms have silent failure modes. "If `$ARGUMENTS` contains `--ci`" is a substring match — it silently fires on `--ci-cd`, `--circular`, or any compound flag sharing the prefix, setting `CI_MODE=true` and bypassing Stage 4's mandatory plan-review gate on an interactive invocation the user did not intend to be CI-gated. "If the first non-whitespace token of `$ARGUMENTS` is exactly `--ci`" is position-constrained — it silently rejects `/roughly:build feature.md --ci` (flag after target argument), breaking a documented and valid invocation pattern. The correct form: "If `$ARGUMENTS` contains `--ci` as a standalone token (preceded by whitespace or string start, followed by whitespace or string end — not as a substring of `--ci-cd` or similar)." This is both substring-safe and position-free. E03.S11b-2 traversed all three iterations in review: (1) substring form caught as P1 silent-failure by the silent-failure-hunter; (2) first-token form caught as P2 ordering-break by `cubic review`; (3) standalone-token form passed all reviewers. Final form is in commit `2e6548d`. **Pattern:** any future skill (build, fix, or later) that detects `$ARGUMENTS`-embedded flags must use the standalone-token form as the default — explicitly bounded by whitespace/string delimiters, not positionally anchored.

- **Decision-table summary lines are discrete emit sites, not restatements of the bullets above them.** A pipeline-skill decision table — "If X: A. If Y: B. If Z: C." at the end of a stage — looks like a recap of the preceding bullets, but an LLM orchestrator may read the summary instead of the bullets. If a prose sweep edits the bullets and leaves the summary untouched, the summary becomes the authoritative (and now stale) emit site: the orchestrator silently emits unstructured messages while the bullets specify structured ones. Concrete origin: E03.S9 Stage 5c in `build/SKILL.md` and `fix/SKILL.md` has three summary lines (spec-compliance pass, spec-compliance fail, quality-check-cap fail). T2/T4 rewrote the first two; the third ("If quality check auto-fix fails after the applicable cap: escalate to human.") was flagged as a "confirm during edit" footnote in the plan and missed. Both code-reviewer and silent-failure-hunter independently caught it at Stage 6 review cycle 1, requiring a review-fix cycle. **Pattern:** when an implementation plan enumerates abort/escalation sites for a sweep, number every line in a decision-table block as a separately numbered edit site — including trailing summary lines. Never leave them as "confirm during edit" notes.

- **Backtick-quoted emit strings followed by additional prose take exactly one period — inside the backtick.** When an inline `` `...message.` `` is followed by a continuation sentence, placing a period inside the backtick AND a period outside it produces `` `...message.`. If next...`` — visually noisy and ambiguous about where the emit string ends. The LLM may include the outer period in the emit, or stop parsing one character early. Concrete origin: E03.S9 `build/SKILL.md` L180 / `fix/SKILL.md` L187 — the auto-fix-cap emit string ended with a period inside the backtick, then was followed by `. If unclear which check produced the failure...`, creating the double-period pattern. Silent-failure-hunter flagged it at review cycle 1. Resolution: drop the outer period; the period inside the closing backtick terminates both the quoted message and the surrounding sentence; the next sentence begins after the closing backtick + space. **Pattern:** when writing an inline emit string followed by additional prose, place exactly one period — inside the closing backtick. No period after the closing backtick.

- **Markdown-list-nested heredocs in copy-paste-ready docs are indentation-fragile and silently break.** When a how-to doc places a `cat > file <<'EOF' ... EOF` heredoc inside a numbered Markdown list, CommonMark renders the code fence with its 3-space list indentation preserved. A copy-paste of the rendered block carries those leading spaces into every line, including the closing `EOF`. Shell heredoc syntax requires the terminator at column 0 (unless `<<-` is used with tabs, which Markdown indentation does not provide), so the heredoc never closes and the user is stranded at `heredoc>`. In E03.S1 T1 (the verification harness doc), this was discovered by the human mid-test and required an in-session patch of two heredoc steps. **Pattern:** copy-paste-ready shell commands inside numbered Markdown lists must avoid heredocs entirely. Use `printf '%s\n' 'line1' 'line2' > file` for multi-line file writes — `printf` is indentation-agnostic at any list nesting level. For JSON or larger files, prefer a single `printf` chain over `cat <<EOF`. Reference: [docs/planning/spikes/s1-verification-harness.md](../docs/planning/spikes/s1-verification-harness.md) Section 4 Steps 3–4 (post-fix); commit c598ef6.

## Documentation Hygiene

- **Appending text to an HTML comment block where `-->` is inline shifts the closing-delimiter line number.** When an existing HTML comment ends with `... last sentence. -->` on a single line and you insert a new sentence on its own line before `-->`, the closing delimiter migrates to a new line number along with the new content. Any sync notes, line-citation comments, or downstream references to "L13" (or wherever `-->` used to be) become stale. Pattern: when citing the line of an HTML-comment-internal note, cite the line containing the new content (where `-->` now lives), not the line where `-->` historically lived. Verify with `rg -n '<!--|-->'` after the edit — must show exactly one of each, and the new line numbers must match what surrounding documentation claims.

- **Prose-only grep filters silently miss rename tokens inside code fences.** When sweeping a file for a rename token (e.g., `Ruckus` → `Roughly`), filtering matches by Markdown context — excluding fenced blocks to "stay in prose" — will skip tokens embedded in code fences, directory trees, sample paths, and comment lines inside fences. In S02.5, `README.md` L287 contained the directory label `ruckus/` inside a `text` code fence; a fence-aware filter that excluded fenced content would surface L1, L3, L20, and others but skip L287 entirely, leaving a stale token in a high-visibility section. The corrective pattern: use plain `rg -n '<token>' <file>` with no Markdown-context filter and review every match individually — fence content is in scope unless an explicit "Migrating from X to Y" carve-out exists. Confirm the rename is complete by re-running the same unfiltered grep and verifying only intentional legacy-name references remain.

- **Append-only edits to immutable documents must use `Edit`, not `Write`, to guarantee zero deletions.** When a task is strictly additive — such as appending a footnote to an ADR — using the `Write` tool with the full new file content risks producing `-` lines in the diff if any prior line is imperceptibly changed (trailing whitespace collapse, line-ending re-encoding, missing final newline). In S02.5 T4, four ADRs each received a v0.1.4 footnote append; a `Write`-based approach would have failed the "zero deletions" acceptance check even if the content was correct. The corrective pattern: use the `Edit` tool with `old_string` set to the file's current last line (read the file first to confirm) and `new_string` set to that same last line followed by `\n\n<new content>` — every preceding line is then byte-identical post-edit. Verify with `git diff <file>` showing only `+` lines and no `-` lines, and `tail -1 <file>` matching the appended content. This pattern generalizes to CHANGELOGs, audit logs, and any append target with immutability constraints.

- **`Edit`'s `replace_all: true` is dangerous when the same token serves dual semantic roles in the same file.** In S02.7 wrap-up (2026-04-30), the target was `.claude/hooks/verify-all.sh`: the intent was to update user-facing comment text at L2 (`ruckus plugin` → `roughly plugin`) and L11 (`ruckus repo` → `roughly repo`). Using `Edit` with `old_string: "ruckus"`, `new_string: "roughly"`, and `replace_all: true` also flipped the legacy-path drift-detection block at L17–L19 — the comment `# Path drift: agents/ should not reference legacy .ruckus/known-pitfalls`, the `rg -q '\.ruckus/known-pitfalls' agents/` grep, and its error-message string all silently became `.roughly/known-pitfalls`. The hook's drift detector now hunted for the current path instead of the legacy path it was designed to catch. The regression was invisible in CI (exit 0) because no agent file triggered the inverted grep; it was only caught by reading L17–L19 with fresh eyes after the edit. Recovery required a surgical `Edit` to restore L17–L19 to `.ruckus/` while leaving L2/L11 in place. The corrective pattern: before using `replace_all: true` for any rename token, scan the file for occurrences where the OLD form is intentional — legacy detection, migration-context strings, historical references, "renamed FROM X" prose — and use surgical `Edit` calls with unique surrounding context to target only the intended lines. This rule generalizes to any bulk-replacement tool (sed -i, IDE find/replace). Verify with `rg -nw 'old-token' <file>` and `rg -nw 'new-token' <file>` together — both outputs should match expectations with no surprises.

- **Pre-existing markdown lint warnings on PostToolUse:Edit hooks are unrelated to the changed lines.** An `Edit` on a single line can trigger dozens of lint warnings in the PostToolUse hook output if the file has pre-existing style violations elsewhere. In E03.S3, editing `docs/ROADMAP.md` L55 surfaced ~30 warnings on L14, L44, L52-73, L92-93, and L115-137 — none touched by the edit. Treating out-of-hunk warnings as "caused by my edit" leads to scope creep or false alarms. Verify by running `git diff <file>` and confirming the warning's line number is inside a `+` or `-` hunk; ignore it if not. Common in older files: `docs/ROADMAP.md`, `docs/planning/epics/*`, `README.md`.

- **Doc claims that cite specific line numbers in cross-referenced files rot silently.** When file A documents "see X at line N of file B," edits to file B (a new check inserted, a block reordered) shift line numbers without triggering any test failure, type error, or grep. The doc claim becomes wrong; readers either distrust correct code or lose faith in the doc. No hook, build step, or verify-all check has awareness of doc-to-code line citations. Concrete origin: `CONTRIBUTING.md` L66–67 (added in E03.S5) cites specific match line numbers in `.claude/hooks/verify-all.sh` ("3 matches at lines 17, 18, 19" and "2 matches at lines 2, 11") to make worked examples reproducible. The silent-failure-hunter review flagged this as a medium-grade rot risk: any structural edit to `verify-all.sh` above line 17 silently invalidates the examples. The E03.S5 AC explicitly required expected match outputs, so line numbers were kept — the rot risk was accepted given the file's low churn rate. **Pattern:** prefer match counts or function/section names over line numbers when citing across files. If an AC requires line numbers (as in E03.S5), document the rot risk and revisit if the cited file's churn rate increases. For high-risk citations, add a self-test that runs the cited grep and asserts the count.

## Planning & Scoping

- **Splitting a doc-only rename from a runtime-directory rename leaves the dogfood repo silently broken between merges.** When an epic partitions paired path renames into separate stories — e.g., S2.3 (agent docs reference `.roughly/known-pitfalls.md`) and S2.6 (this repo's `.ruckus/` → `.roughly/` directory rename) — merging the doc-only story first means every agent in *this dogfood repo* reads from a path that does not exist on disk until S2.6 lands. Claude Code's file read on a missing path returns empty silently, so agents proceed without project context with no error signal. When designing future split renames, either bundle the paired stories into one PR or land them in immediate succession; if they must be sequenced, document the silent-failure window in the commit message so reviewers can decide whether to merge or hold.

- **`permission_mode` in hook stdin is a programmatic plan-mode signal that planners routinely miss.** When E03 was drafted, the team assumed the only signals available to detect Claude Code harness state were prose-based (system-reminder text, tool-availability inference) — readable by the orchestrator model but not by any structured consumer. The E03.S0 spike discovered that hook scripts receive a `permission_mode` field in their stdin JSON with a documented enumerated value of `"plan"` when plan mode is active. This is a stable, programmatic signal at the invocation boundary, unavailable to skill bodies but fully available to hook scripts. Missing this signal during E03 scoping narrowed the mechanism design to prose-only options before the design space was fully understood. **When scoping any story that requires detecting Claude Code harness state (mode, permissions, session metadata), consult the hooks-API stdin schema first.** `permission_mode` and similar structured fields expand the available enforcement points beyond anything readable by the orchestrator model, and omitting them from the design-space survey leads to over-constrained mechanism choices. Reference: [docs/planning/spikes/plan-mode-detection-findings.md](../docs/planning/spikes/plan-mode-detection-findings.md) Section 1 Signal C; commit 9dceb2e.

- **"Confirmed empirically" ACs in spike stories can be unachievable from inside an active pipeline session.** E03.S0 AC2 required that plan-mode auto-engagement triggers be "confirmed empirically, not from documentation alone." The spike was executed inside an active Claude Code build pipeline session — which made it impossible to toggle plan mode, install mode-altering SessionStart hooks, or restart the session without disrupting the pipeline under investigation. Every trigger was documented from the Claude Code permission-modes docs but none was empirically exercised; AC2 was unmet at delivery despite fully reasonable execution. **When writing ACs for spike stories that may run inside a Claude Code session, treat empirical-confirmation requirements as "confirmed empirically OR documented as pending-verification with a clear next-story validation step."** Make the gap-acknowledgment path a first-class option, not a fallback. If empirical confirmation is truly required, either pair the spike with a dedicated out-of-pipeline test session and call this out explicitly in the story's Out of Scope section, or move the empirical work into the follow-on implementation story where an isolated test session can be stood up cleanly. Reference: [docs/planning/spikes/plan-mode-detection-findings.md](../docs/planning/spikes/plan-mode-detection-findings.md) Section 2 AC2 acknowledgment paragraph; [E03.S0 AC2](../docs/planning/epics/E03-trust-and-ergonomics.md) line 71.

- **Spike-doc external API identifiers must be re-verified against the live API at implementation start.** When a spike concludes that a feature depends on a specific external API identifier — a hook event name, a tool name, a JSON field — the spike doc becomes the inherited reference for downstream stories. In E03.S1 (commit c598ef6), [docs/planning/spikes/plan-mode-detection-findings.md](../docs/planning/spikes/plan-mode-detection-findings.md) repeatedly referenced the hook event as `UserPromptExpansion`, a name that does not exist in Claude Code's hooks system. The actual event is `UserPromptSubmit`. The misnomer propagated through three iterations of E03 planning (epic, S0 spike, S1 plan + plan-review PASS), into the verification harness doc, and into `settings.json` snippets; it was first caught only when the human ran the harness against a live Claude Code session and got an empty log. - **Plan `Verify:` commands using `grep -n '<substring>'` are subject to default regex interpretation — use `grep -Fn '...'` for literal-substring verification.** The default `grep` mode treats `.`, `*`, `+`, `[`, `]`, `?`, `^`, `$`, `\`, `|`, `(`, `)`, `{`, `}` as regex metacharacters. A verify pattern like `grep -n 'the ."verify-all'` requires *two* characters between `the ` and `"verify-all` (`.` matches any char, then `"` is matched literally), but the actual text `the "verify-all` has only one. The grep silently returns zero matches even after a correct edit — verifying nothing. Concrete origin: E03 audit-fix T3 plan-write; Stage 4 review-plan subagent caught the regex `.` metachar mismatch before implementation. The fix-plan now uses `grep -Fn '...'` (fixed-string flag with single quotes) for all three verify commands; iteration-2 review-plan PASSed. **Pattern:** when an implementation plan's `Verify:` step is a literal-substring presence check, default to `grep -Fn '<substring>' <file>` (single quotes prevent shell interpretation; `-F` prevents regex interpretation). Reserve `grep -n` for actual regex patterns where every metacharacter is intentional. This applies whether the substring contains obviously-regex chars (`.`, `*`, `?`) or not — the discipline avoids false-negative verify passes on innocuous-looking edits.

- **Grep literal substrings break against bold-decorated markdown field labels.** `grep -Fn "Status: Accepted"` returns 0 hits when the file contains `**Status:** Accepted` — the literal byte sequence is `Status:** Accepted`, not `Status: Accepted`. This is distinct from the metachar pitfall above: the pattern is fine; the haystack decoration silently breaks the match. Fix: include the bold markers in the pattern (`grep -Fn "**Status:** Accepted"`), or grep a substring that lives entirely outside the bold span (`grep -Fn "Status:**"` is safe; `grep -Fn "Status: Accepted"` is not). Concrete origin: E04.S7 T1 verify step; caught by subagent at runtime. **Pattern:** when writing verify-step greps against markdown files with bold field labels (`**Field:** value`), choose a search anchor that does not straddle a bold-close marker.

- **Backport-from-template stories hide completeness gaps when the plan enumerates per-edit changes instead of diffing the final result against the reference.** When a story is "mirror change X from template Y," writing the plan as a discrete change list ("delete line N, modify line M") frames Stage 4 review and Stage 5 verification around "did the enumerated edits land," not "does the result match the template end-to-end." If the template has N coordinated changes and the plan enumerates N-1 of them, every per-edit check passes and the missing change ships. Concrete origin: E04.S4 plan enumerated two edits (drop `set -e`, add `|| true` to `git rev-parse`); the template at `skills/setup/templates/verify-all-stop-hook.sh.template` actually has three coordinated changes — the third is `cd "$ROOT" 2>/dev/null || exit 0`, called out in the template's own comment block as the reason `set -e` is removed ("each potentially-failing command below is explicitly guarded"). Plan-reviewer (cycle 2 PASS), Stage 6 (all three reviewers clean — silent-failure-hunter noted the `cd` regression but classified it negligible), and Stage 7 verify-all all passed; the gap was caught only by post-commit `cubic review`. Two commits were needed: 9e665f2 (the enumerated two edits) and 614fe6a (the missed third edit). **Pattern:** any backport-from-reference story must add an explicit Stage 7 verification step that does `diff <(extract relevant section of target) <(extract relevant section of reference)` and confirms the relevant blocks are byte-identical (or semantically equivalent if placeholders differ). Per-edit checks are insufficient — the verification must operate at the "does the result match the reference" level.
